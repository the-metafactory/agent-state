## Architecture

`agent-state` is the runtime artifact that satisfies the `state` field (formerly
`instanceStateSpec`) of an agent manifest, per [`forge/design/agent-platform.md`](https://github.com/the-metafactory/forge/blob/main/design/agent-platform.md).
It ships as an arc-installable skill bundle and is invoked by hosts (Grove, pilot, ...) at
well-defined lifecycle moments via subprocess invocation.

### Bundle layout

```
agent-state/
├── arc-manifest.yaml         — schema: pai/v1, type: skill, name: AgentState
├── skill/
│   ├── SKILL.md              — trigger phrases + workflow routing
│   ├── Workflows/            — one file per named lifecycle workflow
│   │   ├── ScaffoldFolders.md
│   │   ├── EnqueueWorkItem.md
│   │   ├── ClaimWorkItem.md
│   │   ├── ResolveWorkItem.md
│   │   ├── AppendEvent.md
│   │   ├── ReplayPending.md
│   │   ├── RegenerateDashboard.md
│   │   └── RetrospectiveSummary.md
│   ├── scripts/              — bun-runnable CLIs the workflows shell out to
│   │   ├── scaffold.ts
│   │   ├── errands.ts
│   │   ├── events.ts
│   │   ├── dashboard.ts
│   │   ├── retro.ts
│   │   └── lib/              — shared helpers: db, work-items, events, args
│   └── migrations/
│       └── 0001-initial.sql  — work_items + events tables + indexes
└── __tests__/                — bun test suite (schema, CLI, replay, scaffold)
```

Phase 1 laid down the repo skeleton; Phase 2 shipped the workflows, scripts, migrations,
and tests (as-001 MVP, as-002 scaffold).

### Two SQLite tables

The whole state primitive is **two tables** in `state.sqlite`:

- **`work_items`** — mutable rows representing the agent's queue of work. The `kind` column
  is **agent-defined** (this bundle does not enumerate valid values); the `status` column
  follows the standard lifecycle `pending → in_flight → done | failed | cancelled`, with
  `waiting_human` as a parked state that is never auto-claimed or replayed. Each
  agent decides what `kind` values it cares about (e.g. forge: `release`, `tag`, `announce`;
  luna: `review`, `loop-step`).
- **`events`** — append-only timeline. Every state transition on a work item, every external
  signal received, every workflow invocation writes a row here. Source of truth for the
  dashboard, the retro summarizer, and post-incident audits.

The split is deliberate. `work_items` is a **queue** (mutable, query-by-status). `events` is a
**log** (immutable, query-by-time). Mixing the two leads to either a queue you can't replay
or a log you can't dispatch from.

### Four-folder per-instance layout

Each agent installed into a host gets exactly one directory:

```
~/.config/<host>/agents/<name>/
├── state.sqlite              — the queue + log
├── dashboard.md              — regenerated by RegenerateDashboard, human-readable status
├── retros/                   — RetrospectiveSummary outputs, one file per period
├── context/                  — repos.md + channels.md scope placeholders (scaffolded)
├── CLAUDE.md                 — instance bridge pointing sessions at this state (scaffolded)
└── persona.md                — the agent's persona (copied from bundle by the host on install)
```

The bundle never writes outside this directory. The host owns the path; the bundle owns the
schema and the workflows.

### Workflows

| Workflow | Trigger | Effect |
|----------|---------|--------|
| `ScaffoldFolders` | `arc install` / first run | Creates the per-instance layout, initializes `state.sqlite` from `migrations/0001-initial.sql` |
| `EnqueueWorkItem` | Host `onMessageAccepted` hook | Inserts a row into `work_items` with `status=pending` |
| `ClaimWorkItem` | Worker pulls next item | Atomically transitions a `pending` row to `in_flight`, returns it; emits `work_item_claimed` |
| `ResolveWorkItem` | Work completes | Transitions an `in_flight` row to `done`, `failed`, or `cancelled`; appends terminal event |
| `AppendEvent` | Any observable transition | Inserts an immutable row into `events` |
| `ReplayPending` | Host `onStart` hook | Re-emits `pending` rows and stale `in_flight` rows (past the agent-tunable threshold) for crash recovery |
| `RegenerateDashboard` | `onStateTransition` hook | Rebuilds `dashboard.md` from `work_items` + recent `events` |
| `RetrospectiveSummary` | Operator command / cron | Emits a markdown summary of `events` over a window into `retros/<date>.md` |
